Архитектурный справочник NuBody: Глубокий рефакторинг систем внешности и анатомии

1. Глобальный архитектурный концепт и проблематика

В рамках масштабных пулл-реквестов была проведена полная ликвидация легаси-системы анатомических слотов (OldBody). До рефакторинга строение тел существ в движке описывалось через жесткий иерархический граф соединений слотов внутри легаси-прототипа type: body. Вся визуальная составляющая моба (текстуры, цвета, метки) была намертво привязана к монолитному компоненту внешности (HumanoidAppearanceComponent), а визуальные ресурсы рас жестко дублировались в прототипах type: speciesBaseSprites.

Такой подход имел ряд архитектурных изъянов: он делал невозможной корректную визуальную изоляцию конечностей при хирургическом вмешательстве или расчленении (оторванная рука теряла свой спрайт, цвет кожи и татуировки), усложнял поддержку форков из-за огромного количества дублирующегося YAML-кода и не позволял привязывать визуальные метки к конкретным частям тела (например, шрам только на левой руке).

Новая спецификация NuBody переводит физическое строение существа на рельсы полностью децентрализованного компонентного (ECS) подхода. Тело теперь представляет собой плоский рантайм-набор самостоятельных сущностей органов и конечностей, помещенных в слоты-контейнеры материнской сущности. Каждый отдельный орган содержит собственный визуальный компонент разметки слоев (VisualOrganComponent), а все кастомизируемые маркировки (волосы, узоры, шерсть, татуировки) теперь привязаны непосредственно к конкретной категории анатомического органа (Organ Category Prototype ID).

⚙️ Спецификация изменений на уровне персистенции (База данных EF Core)

Рефакторинг затронул структуру хранения профилей персонажей в БД. Миграции баз данных (например, скрипт 20260118084629_OrganMarkings.cs для PostgreSQL) добавляют в таблицу профилей profile новую колонку organ_markings с типом данных jsonb.

Из структуры десериализации базовых профилей (таких как CharlieCharlieson) полностью удалены устаревшие выделенные примитивные строковые поля цвета и стиля волос, а также цвета и стиля растительности на лице. Теперь волосы и борода не являются какими-то особыми параметрами гуманоида — они сериализуются внутри общего JSON-словаря как стандартные маркировки, накладываемые на зону головы (орган Head). Все существующие учетные записи игроков автоматически конвертируются при первом входе.

2. Критические изменения для разработчиков форков (Breaking Changes)

Данный раздел содержит полный перечень инфраструктурных изменений, которые ломают обратную совместимость. Если вы поддерживаете собственный сервер, форк или кастомные расы, эти изменения необходимо учесть при слиянии (merge) апстрима в ваш репозиторий.

⚠️ Удаление легаси-классов систем и компонентов

Класс HumanoidAppearanceSystem и связанный с ним компонент HumanoidAppearanceComponent полностью удалены из кодовой базы сервера и клиента. Любые вызовы к ним в ваших кастомных системах (медицинские сканеры, импланты, аномалии, системы маскировки шпионов, мутации) вызовут ошибку компиляции. Все ссылки на статические данные профиля должны быть переведены на HumanoidProfileComponent, а управление визуальными слоями — на SharedVisualBodySystem.

⚠️ Краш сервера на этапе инициализации YAML парсера

Следующие типы прототипов были полностью вырезаны из парсера движка RobustToolbox: type: speciesBaseSprites, type: humanoidBaseSprite и type: markingPoints. Если в ваших кастомных конфигурационных YAML-файлах остался хоть один из этих типов (или ссылки на них в параметрах рас), сервер аварийно завершит работу с критической ошибкой YAML load fail до запуска ECS.

⚠️ Удаление синтаксиса Magic Mirror и легаси-пикеров

Старый монолитный XAML-виджет SingleMarkingPicker и логика обработки одиночных сетевых сообщений (например, MagicMirrorAddSlotMessage) полностью ликвидированы. Настройка внешности через парикмахерские зеркала, меню создания персонажа и админ-панели теперь использует унифицированную поорганную модель представления MarkingsViewModel.

3. Спецификация компонентов данных (Content.Shared)

Компонентная модель NuBody абстрагирует описание физического строения и правила отрисовки слоев в изолированные структуры данных.

InitialBodyComponent

Обеспечивает декларативное наполнение родительской сущности дочерними органами в момент ее первичной инициализации через InitialBodySystem. Это избавляет разработчиков от необходимости писать жесткий императивный спавн сущностей через C#-код.

Свойство (C#)Тип данныхYAML тегОписание
Organs Dictionary<ProtoId<OrganCategoryPrototype>, EntProtoId> organs Плоский ассоциативный массив, сопоставляющий уникальную категорию анатомической зоны (ключ) с конкретным ID прототипа сущности органа для спавна (значение).

VisualOrganComponent

Данный компонент вешается непосредственно на сущность органа или конечности. Он служит дескриптором для клиентского VisualBodySystem, указывая, в какой именно графический слой родительского тела необходимо проецировать базовый спрайт данной части тела.

Свойство (C#)Тип данныхYAML тегОписание
Layer string layer Идентификатор целевого рендеринг-слоя на холсте тела (например, базовый текстурный слой кожи).
Data Dictionary<string, string> data Дополнительный словарь метаданных для передачи специфических параметров (цветовых масок, стейтов) в шейдеры.

VisualOrganMarkingsComponent

Указывает рантайм-системам, что данный орган поддерживает динамическое нанесение кастомизируемых пользователем маркировок (шерсть, татуировки, шрамы). Компонент накапливает и синхронизирует сетевое состояние примененных меток между сервером и клиентами.

Свойство (C#)Тип данныхСетевая репликацияОписание
MarkingData OrganMarkingData Static Blueprint Конфигурационная структура, хранящая ссылку на правила группы markingsGroup и перечень допустимых подслоев.
Markings Dictionary<HumanoidVisualLayers, List<Marking>> [AutoNetworked] Рантайм-список фактически нанесенных на текущий момент маркировок, реплицируемый сервером всем клиентам в зоне видимости.

HumanoidProfileComponent (ex. HumanoidAppearanceComponent)

Является абсолютной архитектурной заменой старого монолитного компонента внешности. Однако теперь его зона ответственности строго ограничена хранением метаданных профиля существа (раса, пол, возраст, базовая палитра кожи и глаз). Из структуры компонента полностью вырезаны поля species (список базовых спрайтов расы) и hideLayersOnEquip. Управление лимитами маркировок и динамическим сокрытием слоев одежды полностью делегировано обособленным ECS-системам движка.

4. Реализация клиентских систем рендеринга слоев (Content.Client)

Клиентские системы отвечают за финальный композитинг спрайтов на экране игрока, динамически реагируя на изменения состояния компонентов.

VisualBodySystem.cs

Класс VisualBodySystem наследует базовую логику от SharedVisualBodySystem и управляет послойной сборкой RSI-состояний на холсте SpriteComponent родительской сущности. При инициализации система регистрирует разветвленную сеть обработчиков событий для отслеживания хирургических вмешательств, ампутаций и изменений профиля: 

namespace Content.Client.Body;

public sealed class VisualBodySystem : SharedVisualBodySystem
{
    [Dependency] private readonly IConfigurationManager _cfg = default!;
    [Dependency] private readonly IPrototypeManager _prototype = default!;
    [Dependency] private readonly MarkingManager _marking = default!;
    [Dependency] private readonly SpriteSystem _sprite = default!;

    public override void Initialize()
    {
        base.Initialize();

        // Подписки на физическое внедрение/извлечение (ампутацию) органов
        SubscribeLocalEvent<VisualOrganComponent, OrganGotInsertedEvent>(OnOrganGotInserted);
        SubscribeLocalEvent<VisualOrganComponent, OrganGotRemovedEvent>(OnOrganGotRemoved);
        SubscribeLocalEvent<VisualOrganComponent, AfterAutoHandleStateEvent>(OnOrganState);

        // Подписки на изменение состава кастомизируемых маркировок на конкретном органе
        SubscribeLocalEvent<VisualOrganMarkingsComponent, OrganGotInsertedEvent>(OnMarkingsGotInserted);
        SubscribeLocalEvent<VisualOrganMarkingsComponent, OrganGotRemovedEvent>(OnMarkingsGotRemoved);
        SubscribeLocalEvent<VisualOrganMarkingsComponent, AfterAutoHandleStateEvent>(OnMarkingsState);

        // Трансляция события изменения видимости слоев из-за надетой одежды
        SubscribeLocalEvent<VisualOrganMarkingsComponent, BodyRelayedEvent<HumanoidLayerVisibilityChangedEvent>>(OnMarkingsChangedVisibility);

        // Регистрация колбеков на изменение клиентских и серверных настроек цензуры наготы
        Subs.CVar(_cfg, CCVars.AccessibilityClientCensorNudity, OnCensorshipChanged, true);
        Subs.CVar(_cfg, CCVars.AccessibilityServerCensorNudity, OnCensorshipChanged, true);
    }
}
Алгоритм генерации уникальных ключей слоев

Генерация и наложение графических слоев меток поверх базового спрайта органа выполняются строго по защищенному контракту именования ($"{proto.ID}-{rsi.RsiState}"), что исключает коллизии и конфликты перезаписи индексов в рендерере: 

private void ApplyMarkings(Entity<VisualOrganMarkingsComponent> ent, EntityUid target)
{
    RemoveMarkings(ent, target); // Полная очистка старых зарегистрированных слоев перед ребилдом
    
    int index = 0;
    _sprite.LayerMapTryGet(target, ent.Comp.MarkingData.Layer, out index, true);

    int i = 0;
    foreach (var (marking, alpha) in AllMarkings(ent))
    {
        if (!_marking.TryGetMarking(marking, out var proto)) continue;

        foreach (var rsi in proto.Sprites)
        {
            var layerId = $"{proto.ID}-{rsi.RsiState}";
            if (!_sprite.LayerMapTryGet(target, layerId, out _, false))
            {
                var layer = _sprite.AddLayer(target, sprite, index + i + 1);
                _sprite.LayerMapSet(target, layerId, layer);
                _sprite.LayerSetSprite(target, layerId, rsi);
                
                // Динамическое применение цветовой палитры маркировки, переданной из профиля
                _sprite.LayerSetColor(target, layerId, marking.MarkingColors[i]);
            }
            i++;
        }
    }
}
Подсистема цензуры наготы и автоматического нижнего белья

При изменении CVars цензуры наготы (AccessibilityClientCensorNudity / AccessibilityServerCensorNudity), метод OnCensorshipChanged осуществляет полный принудительный сброс и перерисовку спрайтов. Внутри итератора AllMarkings зашита логика автоматической подстановки дефолтных слоев скрытия наготы. Если игрок не выбрал кастомное нижнее белье, система обращается к конфигурации NudityDefault группы прототипов расы: 

private IEnumerable<Marking> AllMarkings(Entity<VisualOrganMarkingsComponent> ent)
{
    // ... yield return пользовательских маркировок ...

    var censorNudity = _cfg.GetCVar(CCVars.AccessibilityClientCensorNudity) || _cfg.GetCVar(CCVars.AccessibilityServerCensorNudity);
    if (!censorNudity)
        yield break;

    var group = _prototype.Index(ent.Comp.MarkingData.Group);
    foreach (var layer in ent.Comp.MarkingData.Layers)
    {
        if (!group.Limits.TryGetValue(layer, out var layerLimits) || layerLimits.NudityDefault.Count < 1)
            continue;

        var markings = ent.Comp.Markings.GetValueOrDefault(layer) ?? [];
        if (markings.Any(m => _marking.TryGetMarking(m, out var proto) && proto.BodyPart == layer))
            continue; // Пользователь нанес кастомное нижнее белье, дефолт цензуры игнорируется

        // Насильно инжектируем дефолтные спрайты белья, если слой голый
        foreach (var defaultMarkingId in layerLimits.NudityDefault)
        {
            yield return new Marking(defaultMarkingId, 1);
        }
    }
}

Управление динамическим скрытием слоев одежды (HideableHumanoidLayersSystem.cs)

Класс обрабатывает компонент HideableHumanoidLayersComponent. Он заменяет старую разрозненную логику скрытия частей тела при надевании скафандров, громоздкой обуви или закрытых шлемов. Основной метод UpdateSprite работает по принципу отслеживания математической разности состояний (Diff алгоритм) между хэш-сетами HiddenLayers и LastHiddenLayers, минимизируя нагрузку на вызовы рендерера: 

private void UpdateSprite(Entity<HideableHumanoidLayersComponent> ent)
{
    // 1. Возвращаем видимость слоям, которые вышли из зоны скрытия (игрок снял шлем/куртку)
    foreach (var item in ent.Comp.LastHiddenLayers)
    {
        if (ent.Comp.HiddenLayers.ContainsKey(item)) continue;

        var evt = new HumanoidLayerVisibilityChangedEvent(item, true);
        RaiseLocalEvent(ent, ref evt);

        if (_sprite.LayerMapTryGet(ent.Owner, item, out var index, true))
            _sprite.LayerSetVisible(ent.Owner, index, true);
    }

    // 2. Скрываем слои, которые попали в список скрытия в текущем кадре
    foreach (var item in ent.Comp.HiddenLayers.Keys)
    {
        if (ent.Comp.LastHiddenLayers.Contains(item)) continue;

        var evt = new HumanoidLayerVisibilityChangedEvent(item, false);
        RaiseLocalEvent(ent, ref evt);

        if (_sprite.LayerMapTryGet(ent.Owner, item, out var index, true))
            _sprite.LayerSetVisible(ent.Owner, index, false);
    }

    ent.Comp.LastHiddenLayers.Clear();
    ent.Comp.LastHiddenLayers.UnionWith(ent.Comp.HiddenLayers.Keys);
}

5. Архитектура слоя MVVM и Сетка пользовательского интерфейса

Для поддержки нового поорганного деления маркировок старое UI-окно кастомизации было полностью переписано с линейного списка на строгую модель MVVM (Model-View-ViewModel).

MarkingsViewModel.cs

Ядро новой UI-архитектуры. Класс агрегирует данные профиля сущности и транслирует их в виджеты. Изолирует визуальную сетку от прямого вызова систем. Валидирует лимиты расы, ограничения по полу и осуществляет интеллектуальное кэширование цветовой палитры в локальный словарь _previousColors для защиты от случайного сброса настроек игроком при переключении причесок: 

public bool TrySelectMarking(ProtoId<OrganCategoryPrototype> organ, HumanoidVisualLayers layer, ProtoId<MarkingPrototype> markingId)
{
    if (!_prototype.TryIndex(markingId, out var markingProto) || !_organData.TryGetValue(organ, out var organData) || !_organProfileData.TryGetValue(organ, out var profileData))
        return false;

    // Валидация ограничений группы расы и пола
    if (EnforceGroupAndSexRestrictions && !_marking.CanBeApplied(organData.Group, profileData.Sex, markingProto))
        return false;

    _markings[organ] = _markings.GetValueOrDefault(organ) ?? [];
    var organMarkings = _markings[organ];
    organMarkings[layer] = organMarkings.GetValueOrDefault(layer) ?? [];
    var layerMarkings = organMarkings[layer];

    // Кэширование: восстанавливаем старые цвета, если игрок уже выбирал эту метку ранее в этой сессии
    var colors = _previousColors.GetValueOrDefault(markingId) ??
                 MarkingColoring.GetMarkingLayerColors(markingProto, profileData.SkinColor, profileData.EyeColor, layerMarkings);
    var newMarking = new Marking(markingId, colors);

    if (!_prototype.TryIndex(organData.Group, out var groupPrototype) || !groupPrototype.Limits.TryGetValue(layer, out var limits) || !EnforceLimits)
    {
        layerMarkings.Add(newMarking);
        MarkingsChanged?.Invoke(organ, layer);
        return true;
    }

    // Автоматическая замена элемента (например, прически) при жестком лимите слоя равном 1
    if (limits.Limit == 1 && layerMarkings.Count == 1)
    {
        layerMarkings.Clear(); 
        layerMarkings.Add(newMarking);
        MarkingsChanged?.Invoke(organ, layer);
        return true;
    }

    if (layerMarkings.Count < limits.Limit)
    {
        layerMarkings.Add(newMarking);
        MarkingsChanged?.Invoke(organ, layer);
        return true;
    }
    return false;
}

Математика Drag & Drop позиционирования слоев (LayerMarkingOrderer.xaml.cs)

Для интерактивного изменения порядка отображения слоев (например, перетащить татуировку поверх слоя шерсти) используется глубокая интеграция со вспомогательным классом DragDropHelper<LayerMarkingDragged>. Во время движения курсора мыши система производит расчёт квадрата расстояния до потенциальных маяков drop-зон (LayerDragDropBeacon). Использование квадрата длины вектора LengthSquared() применяется для оптимизации производительности рендера (исключает тяжелую математическую операцию извлечения квадратного корня на каждый кадр): 

private bool OnContinueDrag(float frameTime)
{
    var (item, container) = _dragDropHelper.Dragged;
    LayoutContainer.SetPosition(item, UserInterfaceManager.MousePositionScaled.Position - new Vector2(32, 32));

    // Поиск ближайшего маяка drop-зоны через квадрат длины вектора
    var closestBeacon = _beacons.MinBy(beacon => 
        (UserInterfaceManager.MousePositionScaled.Position - beacon.GlobalPosition).LengthSquared());

    if (closestBeacon != _dragTarget)
    {
        _dragTarget?.UnbecomeTarget(); // Возврат старого таргета в дефолтное состояние (размер float.NaN)
        _dragTarget = closestBeacon;
        _dragTarget?.BecomeTarget();   // Раскрытие новой зоны сброса: принудительная высота 64px и стиль .PanelDropTarget
    }
    return true;
}

6. Справочник Интеграционных Тестов (MarkingManagerTests.cs)

Для верификации корректности работы миграций был разработан комплекс автоматических интеграционных тестов в модуле MarkingManagerTests.cs. Наиболее важным является тест HairConversion, проверяющий автоматическую конвертацию легаси-причесок гуманоидов (старый плоский массив) в маркировки органа головы новой архитектуры NuBody: 

[Test]
public async Task HairConversion()
{
    await using var pair = await PoolManager.GetServerClient();
    var server = pair.Server;
    await server.WaitIdleAsync();

    await server.WaitAssertion(() =>
    {
        var markingManager = server.ResolveDependency<MarkingManager>();
        // Создаем старую легаси-маркировку волос
        var markings = new List<Marking>() { new("HumanHairLongBedhead2", new List<Color>() { Color.Red }) };

        // Имитируем загрузку старого профиля, проверяя работу конвертера
        var converted = markingManager.ConvertMarkings(markings, "Human");

        // Утверждаем, что волосы теперь принадлежат сущности изолированного органа Head
        Assert.That(converted, Does.ContainKey(new ProtoId<OrganCategoryPrototype>("Head")));
        Assert.That(converted["Head"], Does.ContainKey(HumanoidVisualLayers.Hair));
        
        var hairMarkings = converted["Head"][HumanoidVisualLayers.Hair];
        Assert.That(hairMarkings, Has.Count.EqualTo(1));
        Assert.That(hairMarkings[0].MarkingId, Is.EqualTo("HumanHairLongBedhead2"));
    });
    await pair.CleanReturnAsync();
}

7. Энциклопедия миграции YAML-прототипов и маппинг (Контракты)

Все кастомные YAML-файлы сущностей и рас, написанные под старую архитектуру, должны быть переформатированы. Оставление старых полей приведет к падению парсера.

Архитектурное сравнение: Старый прототип vs Новый прототип

Ниже приведено наглядное сравнение того, как описывалось тело моба и параметры расы до рефакторинга, и как они должны быть описаны теперь.

СТАРЫЙ ПРОТОТИП (Сломает сервер)

1. Конфигурация моба (сложный граф)

- type: entity
  id: MobHuman
  components:
  - type: Body
    prototype: Human # Ссылка на удаленный type: body
    requiredLegs: 2
  - type: HumanoidAppearance # Удаленный компонент

2. Легаси-прототип графа (type: body)

- type: body
  id: Human
  slots:
    torso:
      connections: [ head, left_arm, right_arm ]
    head:
      connections: [ eyes, brain ]
      # Очень длинная иерархия...

3. Легаси-прототип расы

- type: species
  id: Human
  sprites: speciesBaseSpritesHuman # УДАЛЕНО
  markingLimits: markingPointsHuman # УДАЛЕНО
  maleLastNames: [ Smith, Doe ] # УДАЛЕНО
  femaleLastNames: [ Smith, Doe ] # УДАЛЕНО
  dollPrototype: MobHumanDummy
НОВЫЙ ПРОТОТИП (NuBody)

1. Конфигурация моба (Плоский словарь ECS)

- type: entity
  id: MobHuman
  components:
  - type: Body # Оставлен пустым для логики ранений
  - type: InitialBody
    organs:
      Torso: OrganHumanTorso
      Head: OrganHumanHead
      Brain: OrganHumanBrain
  - type: HumanoidProfile # Заменил HumanoidAppearance

2. Иерархия markingsGroup (Новый тип)

- type: markingsGroup
  id: Human
  parent: Undergarments
  limits:
    enum.HumanoidVisualLayers.Hair:
      limit: 1
    enum.HumanoidVisualLayers.FacialHair:
      limit: 1

3. Обновленный прототип расы

- type: species
  id: Human
  # Спрайты и лимиты полностью удалены
  lastNames: [ Smith, Doe ] # Единый массив
  dollPrototype: AppearanceHuman # Префикс изменен

Карта наследования групп маркировок (markingsGroup Hierarchy)

Все расы и их форки обязаны мапиться на валидные корневые и дочерние группы маркировок. Корневым системным элементом для всех рас является абстрактная группа Undergarments (нижнее белье). 

None (Root System)
└── Undergarments (Базовая группа скрытия наготы)
    ├── Human     <-- Целевой маппинг для расы Human
    ├── Vulpkanin <-- Сюда же принудительно перенаправляется форк CorvaxVulpkanin
    ├── Arachnid
    ├── Diona
    ├── Moth
    ├── Reptilian
